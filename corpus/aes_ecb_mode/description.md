# CRYPTO003: использование AES в режиме ECB

## Кратко

В Python есть две распространённые библиотеки симметричной криптографии: cryptography (проект pyca/cryptography, низкоуровневый API в подмодуле hazmat) и pycryptodome (форк pycrypto, импортируется как Crypto.Cipher.AES). Обе позволяют создать AES-шифр в режиме ECB — у PyCA это вызов Cipher(algorithms.AES(key), modes.ECB(), ...), у pycryptodome — AES.new(key, AES.MODE_ECB). Режим Electronic Codebook шифрует каждый 16-байтовый блок открытого текста независимо от других, без связи между блоками и без аутентификации. На практике это почти всегда уязвимость. Правило CRYPTO003 находит все основные формы вызова AES-ECB в обеих библиотеках и предлагает замену на AES-GCM — режим, рекомендованный современной криптографической практикой.

## Почему это проблема

ECB сохраняет на шифротексте структуру открытого текста. Любые два одинаковых блока открытого текста под одним ключом дают один и тот же блок шифротекста, поэтому повторы в данных видны атакующему без знания ключа. Каноническая иллюстрация — пиксельная картинка логотипа Tux: после ECB-шифрования контуры остаются различимыми, потому что соседние одинаковые пиксельные блоки превращаются в одинаковые шифро-блоки. Это свойство ECB, а не свойство AES — оно сохраняется при любом размере ключа и любой реализации.

Режим не удовлетворяет современным требованиям к симметричной криптографии:

- Нет диффузии между блоками. CBC связывает каждый блок с предыдущим через XOR, CTR/GCM — через счётчик, ChaCha20 — через состояние потокового шифра. Только ECB обходится без связи, и именно из-за этого сохраняет структуру открытого текста.
- Нет аутентификации. ECB не проверяет целостность сообщения. Любая модификация байтов на лету будет принята при расшифровке как валидная, даже если результат — мусор. Для аутентификации нужна либо AEAD-обёртка (GCM, ChaCha20-Poly1305), либо отдельный MAC.
- Нет привязки к контексту шифрования. Один и тот же блок открытого текста, зашифрованный под одним ключом в разных сообщениях, даст один и тот же шифротекст. Это позволяет атакующему распознавать повторяющиеся данные между сессиями.

NIST SP 800-38A (Recommendation for Block Cipher Modes of Operation) формально определяет ECB как один из пяти режимов AES, но в современных профилях (TLS 1.2+, JWE, IPsec ESP) ECB не используется. Уязвимость классифицирована как CWE-327 (Use of a Broken or Risky Cryptographic Algorithm).

Сама документация PyCA Hazmat в разделе про modes.ECB содержит явное предупреждение: «ECB is not authenticated and is therefore not recommended; in almost all cases, you should use a different mode such as AES-GCM». Это позиция авторов библиотеки, а не наше частное мнение.

## Последствия по типам использования

ECB опасен во всех типичных сценариях шифрования в backend-коде. Поэтому правило CRYPTO003 не различает контекст и всегда выдаёт severity high — в отличие от CRYPTO001/CRYPTO002, где есть градация. Конкретные сценарии и последствия:

- Шифрование payload в HTTP/RPC. Повторяющиеся блоки в JSON (одинаковые ключи, одинаковые числовые поля у соседних записей) проявляются повторяющимися блоками в шифротексте. Атакующий извлекает структуру сообщения без знания ключа — например, видит, что определённое поле принимает одно из двух значений, и считает по этому статистику.
- Шифрование полей в базе данных. Одинаковые значения у разных пользователей дают одинаковые шифро-значения. Это ломает свойство «шифрование скрывает значение поля» — атакующий с доступом к таблице может группировать пользователей по равенству шифро-значений, эффективно проводя equality-атаку без расшифровки.
- Шифрование сессионных токенов и кук. Block-rearrangement атака: атакующий может переставить или подменить отдельные блоки шифротекста, и расшифровка даст синтаксически валидный, но семантически подменённый результат. В авторизационной куке это может означать смену user_id или роли.
- Шифрование длинных бинарных данных (файлы, изображения). Прямая визуализация уязвимости — Tux. Применительно к реальным данным это означает, что структура файла (заголовки, повторяющиеся секции) остаётся различимой в шифротексте.

Применимая ниша ECB крайне узкая: шифрование одного блока чисто случайных данных нужной длины (например, обёртка ключа другого шифра). В backend-коде встречается она почти никогда — обычно вместо этого используется dedicated key-wrap режим (AES-KW, RFC 3394), который безопаснее.

Дополнительный фактор риска — широкое распространение в туториалах. ECB — это самая простая схема, и первые попавшиеся примеры на Stack Overflow и в туториалах десятилетней давности часто используют именно его. Без статического анализа такие места живут в проекте годами без явного индикатора проблемы.

## Безопасные альтернативы

Современная практика для шифрования произвольных данных в backend-приложении на Python — AEAD-режим, в первую очередь AES-GCM:

- **AES-GCM** (Galois/Counter Mode). AEAD-режим: одновременно шифрует и аутентифицирует, без отдельного MAC. Стандартизован в NIST SP 800-38D, входит в TLS 1.2/1.3, JWE, IPsec. Требует уникального nonce на каждое шифрование под одним ключом (повтор ломает шифр и схему аутентификации). У PyCA — `cryptography.hazmat.primitives.ciphers.aead.AESGCM`, у pycryptodome — `AES.new(key, AES.MODE_GCM, nonce=...)`. Для большинства задач — это правильный выбор по умолчанию.
- **ChaCha20-Poly1305**. AEAD-режим на основе потокового шифра ChaCha20 и MAC Poly1305. Стандартизован в RFC 8439. Альтернатива AES-GCM для платформ без аппаратной поддержки AES; на машинах без AES-NI обычно быстрее. У PyCA — `cryptography.hazmat.primitives.ciphers.aead.ChaCha20Poly1305`.
- **AES-CBC + HMAC** (encrypt-then-MAC). Legacy-схема для случаев, где AEAD недоступен. Требует уникального непредсказуемого IV и отдельного HMAC-ключа, отличного от ключа шифрования. Соблюсти все требования вручную сложнее, чем взять AEAD; типичные ошибки — повтор IV, MAC-then-encrypt вместо encrypt-then-MAC, тот же ключ для шифрования и MAC.
- **AES-CTR + HMAC**. Аналогичная схема со счётчиковым режимом; те же требования к уникальности счётчика и отдельному ключу аутентификации.

Suggestion правила выбран один — AES-GCM, потому что в 95% сценариев backend-кода это правильный ответ. Расширенные сценарии разработчик найдёт в документации сам; задача линтера — не учить криптографии, а указать на проблему и предложить безопасный default.

## Зачем создаётся правило

Существующие инструменты задачу решают неполно. Эмпирическое обоснование на расширенном кейсе из 5 сценариев (две библиотеки, пять форм вызова), даты прогонов 2026-04-29:

- **Bandit 1.9.4** ловит только формы PyCA Hazmat: 2/5 TP, 3 FN, 0 FP по теме ECB. Срабатывает правило B305 (`b304-b305-ciphers-and-modes`) с severity MEDIUM на вызовы `cryptography.hazmat.primitives.ciphers.modes.ECB`. На все три формы pycryptodome (`AES.MODE_ECB` позиционно, `mode=AES.MODE_ECB` kwarg, импорт с алиасом) — пропускает. Систематический пробел: B305 устроен как чёрный список вызовов конкретного класса, а в pycryptodome режим — это не самостоятельный класс, а целочисленная константа атрибута класса AES, и в чёрный список она не попадает.
- **Semgrep 1.161.0** на open-source наборе semgrep/semgrep-rules @ fdc73542 (директория `python/`, 371 правило): 0/5 TP, 5 FN, 0 FP. Полный пробел набора по этой теме. Поиск по `ECB` и `MODE_ECB` в директории `python/` ничего не находит — правила про режим ECB нет ни для одной из двух библиотек.

Дополнительно Bandit B413 (LOW severity) флагует сам факт импорта pycryptodome — это посторонняя находка по другой теме (deprecated библиотека), не относящаяся к выбору режима шифрования; не TP и не FP по теме ECB. Зафиксирована в `analogs.md` отдельно для полноты.

Это **двойной пробел разной природы**: ось 1 (полное отсутствие у Semgrep) и ось 3 (асимметрия у Bandit между PyCA и pycryptodome). Кейс классифицирован как **продвинутый** — по этим двум осям одновременно.

Правило CRYPTO003 закрывает оба пробела одним правилом и добавляет несколько улучшений сверх Bandit на тех формах, которые B305 покрывает:

- Покрытие двух библиотек одной семантикой («режим, переданный AES-шифру, — ECB»). PyCA Hazmat (формы `modes.ECB()` и прямой `ECB()`) и pycryptodome (`AES.MODE_ECB` позиционно, kwarg `mode=`, алиас импорта) детектируются единообразно. Bandit покрывает 2 из 5 форм, Semgrep — 0 из 5, наше правило — 5 из 5.
- Severity high (всегда), без шкалы. Bandit для тех же двух форм PyCA даёт MEDIUM. ECB — бинарное состояние, промежуточного «менее опасного» ECB не существует, поэтому шкала избыточна и затрудняет триаж.
- Информативное сообщение с CWE-327 и пометкой библиотеки, на которой поймана форма (`PyCA Hazmat` или `pycryptodome`). У Bandit сообщение фиксированное и не указывает CWE прямо в выводе.
- Конкретное предложение замены — AES-GCM с указанием API обеих библиотек (`cryptography.hazmat.primitives.ciphers.aead.AESGCM` и `AES.new(key, AES.MODE_GCM, nonce=...)`). У Bandit suggestion отсутствует.

Известные ограничения MVP:

- Если режим передан через переменную (`m = AES.MODE_ECB; AES.new(key, m)`), правило молчит. Без потокового анализа отличить безопасный конфиг (`MODE = AES.MODE_GCM`) от опасного (`MODE = AES.MODE_ECB`) невозможно, и срабатывание на «неизвестно» гарантированно даст FP в реальном коде. Расширение — отдельный кейс с потоковым анализом.
- Импорт `from cryptography.hazmat.primitives.ciphers.modes import ECB as <alias>` с произвольным алиасом учитывается через alias mapping (фактическое имя берётся из `ImportFrom.names[].asname`) — это работает.
- Сценарий «динамический выбор» через тернарник (`AES.new(key, AES.MODE_ECB if cond else AES.MODE_GCM)`) корректно срабатывает: правило обходит AST-узел `AES.MODE_ECB` внутри тернарника и фиксирует, что один из путей опасен.
- Сторонние реализации AES (например, чистый Python через pyaes) — отдельный кейс с другим набором импортов; в текущее правило не входит.
