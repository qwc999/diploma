# CRYPTO004 — bcrypt_low_rounds

## Какая проблема

Использование bcrypt с cost factor ниже безопасного минимума для хеширования паролей. В Python это обычно задаётся параметром `rounds` у `bcrypt.gensalt`: чем меньше значение, тем дешевле один расчёт bcrypt-хеша и тем быстрее атакующий перебирает пароли после утечки базы.

В коде уязвимость выглядит так:

```
bcrypt.gensalt(rounds=4)
bcrypt.gensalt(rounds=8)
gensalt(rounds=10)
```

Для правила CRYPTO004 принят рабочий порог проекта, согласованный с
актуальной практикой OWASP Password Storage Cheat Sheet:

- `rounds >= 12` — безопасно;
- `rounds < 10` — грубо слабое значение, severity high;
- `10 <= rounds < 12` — устаревшее пограничное значение, severity medium.

`bcrypt.gensalt()` без явного `rounds` правило не флагует: для MVP это считается безопасным или неизвестным случаем, чтобы не плодить false positive. Для нового production-кода безопасный вариант — явно задать `rounds=12` или выше и выбрать конкретное значение по latency budget сервиса.

CWE: CWE-916 (Use of Password Hash With Insufficient Computational Effort).

## Почему это проблема

bcrypt — password hashing схема, а не обычная хэш-функция. Её защитная сила держится на том, что проверка каждого кандидата пароля намеренно дорогая. Параметр `rounds` задаёт логарифмическую стоимость вычисления: увеличение на 1 примерно удваивает объём работы. Поэтому разница между `rounds=8` и `rounds=12` — не четыре процента и не четыре операции, а примерно шестнадцатикратная разница в цене перебора.

При утечке базы паролей атакующий получает bcrypt-хеши и соли пользователей. Соль защищает от радужных таблиц и переиспользования одной работы для всей базы, но не делает каждую попытку дорогой. Если cost factor низкий:

- `rounds=4` или `rounds=8` превращают bcrypt в слишком дешёвую проверку для современного железа;
- словарные атаки и перебор популярных мутаций становятся существенно быстрее;
- слабые пользовательские пароли восстанавливаются с заметно меньшей стоимостью;
- ошибка сохраняется годами, потому что старые хеши обычно живут вместе с базой и не пересчитываются автоматически.

Для backend-приложений это особенно важно: password hashes — долгоживущий актив. Низкое значение в одном релизе может остаться в production надолго, даже если код позже будет исправлен.

## Почему это решаем именно мы

Bandit 1.9.4 и Semgrep 1.161.0 (open-source `semgrep-rules` @ `fdc73542`, локальный набор `python/`) на финальном корпусе из 4 сценариев не флагуют ни одного срабатывания — 0/4 TP у обоих, 4 FN у обоих, 0 FP на safe-файле. Причина систематическая: ни один из инструментов не реализует семантику «сравнить значение `rounds` у `bcrypt.gensalt` с порогом».

Это тот же класс пробела, что у CRYPTO002: аналоги могут проверять отдельные криптографические API, но не кодируют численные требования к параметрам password hashing. Поэтому CRYPTO004 вместе с CRYPTO002 образует пару правил «KDF/password hashing parameter checks».

Правило полезно для дипломного линтера по трём причинам:

- закрывает полный двойной пробел двух распространённых анализаторов;
- опирается на понятный числовой порог, который легко объяснить разработчику;
- покрывает реалистичный backend-сценарий: bcrypt часто используется для хранения паролей, а низкие значения `rounds` встречаются в старом коде и демонстрационных примерах.

В отличие от CRYPTO001, где один из аналогов уже ловит сам паттерн и наш вклад в основном в severity/UX, CRYPTO004 закрывает именно пробел детектирования. В этом смысле он ближе к CRYPTO002: оба правила проверяют числовой параметр password hashing API, который конкуренты не анализируют.

В классификации проекта это продвинутый кейс: все аналоги пропустили ожидаемые уязвимые строки.

## Что детектирует правило

Поддерживаются четыре формы вызова `gensalt`:

1. `bcrypt.gensalt(rounds=N)` после `import bcrypt`
2. `bc.gensalt(rounds=N)` после `import bcrypt as bc`
3. `gensalt(rounds=N)` после `from bcrypt import gensalt`
4. `gs(rounds=N)` после `from bcrypt import gensalt as gs`

Severity:

- high — `rounds < 10`;
- medium — `10 <= rounds < 12`;
- срабатывания нет — `rounds >= 12`.

Сообщение правила указывает фактическое значение `rounds`, минимальный порог 12 и CWE-916. Suggestion предлагает поднять cost factor минимум до 12, а для новых production-систем выбирать максимальное значение, которое укладывается в latency budget.

Безопасные формы из корпуса:

```
bcrypt.gensalt(rounds=12)
bcrypt.gensalt(rounds=14)
gensalt(rounds=12)
gs(rounds=14)
bcrypt.gensalt()
```

Известные ограничения MVP:

- если `rounds` задан переменной, а не числовым литералом, правило молчит. Без потокового анализа невозможно отличить безопасный конфиг (`ROUNDS = 12`) от опасного (`ROUNDS = 4`), и срабатывание на неизвестное значение создаст false positive;
- позиционная форма `bcrypt.gensalt(8)` в текущий корпус не включена. Она может быть добавлена отдельным расширением правила, если понадобится;
- `bcrypt.gensalt()` без аргументов не флагуется.
