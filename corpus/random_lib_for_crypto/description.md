# CRYPTO001: использование модуля random в криптографических целях

## Кратко

В Python есть стандартный модуль random для генерации псевдослучайных чисел. Разработчики часто используют его, чтобы сгенерировать токен сессии, ключ, IV для шифра, nonce, соль для хеширования пароля или приватное число. Это небезопасно. Для криптографии нужны функции из модуля secrets либо os.urandom — они опираются на криптостойкий системный источник случайности. Правило CRYPTO001 находит вызовы опасных функций модуля random в коде и предлагает корректную замену.

## Почему это проблема

Модуль random использует алгоритм Mersenne Twister (MT19937). У него прекрасные статистические свойства и огромный период, но он не является криптографически стойким:

- Внутреннее состояние MT19937 — массив из 624 32-битных слов. Зная 624 последовательных 32-битных вывода, злоумышленник полностью восстанавливает состояние через инверсию tempering. После этого предсказуемы все будущие выводы и существенная часть прошлых. Атака публично известна и реализуется готовыми библиотеками вроде randcrack.
- Modul не удовлетворяет требованиям NIST SP 800-90A к криптографическим DRBG: нет backtracking resistance, нет prediction resistance, нет периодического реседирования из независимого источника энтропии.
- Сама документация Python прямо предупреждает: «The pseudo-random generators of this module should not be used for security purposes. For security or cryptographic uses, see the secrets module». Это позиция авторов стандартной библиотеки, а не наше частное мнение.
- Уязвимость классифицирована как CWE-338 (Use of Cryptographically Weak Pseudo-Random Number Generator).

## Последствия по типам использования

Контекст определяет, насколько серьёзны последствия. Поэтому правило CRYPTO001 повышает severity до high, если результат вызова используется как крипто-значение, и оставляет medium иначе.

- Токены сессий, API-ключи, password reset tokens. Предсказуемость позволяет захватить чужую сессию или сбросить пароль чужому пользователю без знания старого.
- IV для AES-CBC. Предсказуемый IV открывает атаку BEAST на CBC; повтор IV ведёт к утечке информации о plaintext.
- Nonce для AEAD-шифров (AES-GCM, ChaCha20-Poly1305). Повтор nonce полностью ломает шифр: восстанавливается XOR двух открытых текстов, для GCM вычисляется ключ аутентификации H, после чего forgery становится тривиальным.
- Соль для парольного хеша. Если соль предсказуема, теряется свойство уникальности соли, и атаку можно вести параллельно по всей базе пользователей.
- Простые числа для RSA или k для ECDSA. Если число предсказуемо, восстанавливается приватный ключ.

Исторические инциденты, иллюстрирующие масштаб: Netscape SSL (1995, Goldberg и Wagner), Debian OpenSSL (2008, CVE-2008-0166, около миллиона уязвимых ключей), Sony PS3 ECDSA (2010, fail0verflow), периодические потери средств в Bitcoin-кошельках, использовавших random для генерации приватных ключей.

## Безопасные альтернативы

В Python для криптографии используется один из следующих источников случайности:

- secrets — модуль стандартной библиотеки, добавленный в Python 3.6 специально для криптографических нужд.
- os.urandom(n) — низкоуровневая обёртка над системным CSPRNG.
- random.SystemRandom() — обёртка с интерфейсом random, под капотом использует os.urandom.

Под капотом эти источники опираются на системный CSPRNG: getrandom() или /dev/urandom на Linux, BCryptGenRandom на Windows, getentropy() на macOS и BSD. Эти генераторы спроектированы с расчётом на криптографическую стойкость и периодически реседируются из независимых источников энтропии ядра.

## Зачем создаётся правило

Существующие инструменты задачу частично решают, но с пробелами:

- Bandit ловит вызовы опасных функций модуля random (правило B311) и для нашего корпуса находит все 6 ожидаемых строк. Однако severity у Bandit во всех случаях одинаковая — LOW, без различения крипто-контекста, и без подсказки чем именно заменить вызов.
- Semgrep в open-source наборе semgrep-rules под директорией python не содержит правила про модуль random — на нашем корпусе 0 срабатываний из 6 ожидаемых.

Правило CRYPTO001 закрывает кейс с тремя улучшениями над Bandit:

- Контекстная severity. Если результат присваивается переменной с крипто-именем (token, key, iv, nonce, secret, password, salt, session_id, api_key) или передаётся как аргумент функции с таким именем, severity повышается до high. Иначе остаётся medium. Это даёт приоритезацию при триаже на больших репозиториях, где LOW-предупреждения от Bandit обычно игнорируются.
- Сообщение с привязкой к контексту. Вместо обобщённого «Standard pseudo-random generators are not suitable...» правило выдаёт «Weak random for cryptographic purposes: value used as token — random.choice is not cryptographically secure», упоминая конкретное крипто-имя из контекста.
- Конкретное предложение замены под каждую функцию: random.choice → secrets.choice(seq), random.randbytes(n) → secrets.token_bytes(n), random.randint(a, b) → a + secrets.randbelow(b - a + 1), random.getrandbits(k) → secrets.randbits(k) и так далее.

Кейс классифицирован как базовый: хотя бы один аналог (Bandit) покрывает его. Уникальный вклад CRYPTO001 на этом кейсе — UX и точность, а не закрытие пробела детектирования. Закрытие пробелов — задача продвинутых правил, которые описываются в отдельных кейсах.
